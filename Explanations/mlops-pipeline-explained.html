What Is MLOps?

MLOps (Machine Learning Operations) is the set of practices that combines ML development with software engineering and DevOps to deploy, monitor, and maintain ML models in production reliably. It covers everything that happens after you train a model: tracking experiments, optimizing hyperparameters systematically, versioning models, packaging them for deployment, serving predictions through APIs, and monitoring performance over time.

In industry, training a model is roughly 20% of the work. The other 80% is the system around it. Companies don't hire you to run model.fit() in a notebook. They hire you to build reproducible, trackable, deployable ML systems. That's what MLOps is, and that's what this project demonstrates.

Why It Matters

Every other project in this portfolio answers "can I train a good model?" This project answers a different question: "can I put a model into production?" MLOps skills are the #1 gap companies complain about in junior ML candidates. Knowing how to use MLflow, Optuna, and FastAPI together sets you apart from candidates who can only show Jupyter notebooks.

The Dataset: Telco Customer Churn
7,043 customers, 26.5% churn rate, 21 features
▼

The Telco Customer Churn dataset from Kaggle contains 7,043 telecom customers with information about their demographics, services, contract type, billing, and whether they churned (left the company). The churn rate is 26.5%, making this an imbalanced binary classification problem.

EDA revealed two strong churn signals: customers who churn tend to have higher monthly charges and shorter tenure (newer customers leave faster). Three new features were engineered: AvgChargesPerMonth (total charges divided by tenure), TenureBin (grouping tenure into 4 buckets), and HasMultipleServices (phone + internet).

The preprocessing pipeline uses sklearn's ColumnTransformer to apply StandardScaler to 6 numeric features and OneHotEncoder to categorical features, ensuring train and test data are always processed identically. This pipeline is bundled with the model when saved, preventing the #1 production ML bug: preprocessing mismatches between training and inference.

MLflow: Tracking Every Experiment
165 total runs logged with params, metrics, and model artifacts
▼

MLflow is an open-source platform for managing the ML lifecycle. Every time a model is trained in this notebook, MLflow automatically logs the hyperparameters, evaluation metrics (accuracy, F1, AUC, precision, recall), training time, and the serialized model artifact. By the end of the notebook, 165 runs are tracked and queryable.

📌 Why This Matters: Without MLflow, you end up with notebooks named "model_v2_final_FINAL_v3.ipynb" and no idea which hyperparameters produced which results. With MLflow, every single experiment is recorded. When someone asks "why did you choose these hyperparameters?", you can pull up the exact run with one command.

Optuna: Smart Hyperparameter Optimization
50 trials, 8 hyperparameters, Bayesian search
▼

Optuna uses Bayesian optimization to search for the best hyperparameters. Unlike grid search (which tries every combination) or random search (which guesses), Optuna learns from previous trials to focus on promising regions of the search space. Each of the 50 trials was also logged to MLflow.

Optuna improved XGBoost's F1 by +1.99% over the baseline, surpassing even Logistic Regression. The optimization also revealed that max_depth=3 (shallow trees) with high regularization (reg_alpha=9.77) works best, preventing overfitting on this relatively small dataset.

Why Optuna on XGBoost and Not Logistic Regression?

This was a deliberate engineering decision, not an arbitrary choice. Logistic Regression has essentially two tunable parameters (C and penalty). You don't need 50 Bayesian optimization trials to search a 2-parameter space. A simple grid search with 10 values of C would cover it in seconds.

XGBoost, on the other hand, has 8+ interacting hyperparameters. These interact in complex ways: low learning_rate needs more n_estimators, high max_depth needs more regularization, subsample and colsample_bytree jointly control variance. That's where Optuna's Bayesian search adds real value because it learns which parameter combinations work well together.

The results validated this decision. Baseline LogReg (F1=0.5889) was already near its ceiling with default settings because the relationship is approximately linear after preprocessing. Baseline XGBoost (F1=0.5748) was underperforming with defaults but had significant room to improve. After tuning, XGBoost jumped to F1=0.5947 and surpassed LogReg.

🔑 Key Insight from Optuna: The parameter importance analysis showed which hyperparameters matter most. This is more valuable than just the best values because it tells you where to focus tuning effort in future projects.

Summary: Key Takeaways
▼
🎯 Takeaway #1: MLflow Solves "Which Experiment Was Best?" 165 runs were logged with full parameters, metrics, and model artifacts. The MLflow dashboard lets you compare, filter, and reproduce any experiment instantly.
🎯 Takeaway #2: Optuna Replaces Guesswork with Science. Bayesian optimization found hyperparameters that improved XGBoost by +1.99% F1 over the baseline. It also revealed which parameters matter most (parameter importance analysis).
🎯 Takeaway #3: Pipelines Prevent Production Bugs. Bundling preprocessing + model into one serialized Pipeline ensures the API applies the exact same transformations as training. No separate preprocessing code, no mismatch risk.
🎯 Takeaway #4: FastAPI Makes Models Usable. A REST API with auto-generated Swagger docs means any developer can send JSON and get predictions back. This is exactly how ML models are deployed at companies.
🎯 Takeaway #5: The Full Lifecycle Matters. Training a model is ~20% of production ML. This project demonstrates the other 80%: data preprocessing, experiment tracking, systematic optimization, model versioning, and API serving.
🎯 Takeaway #6: Simple Models Can Win. Logistic Regression (F1=0.5889) nearly matched optimized XGBoost (F1=0.5947) on this dataset. The right preprocessing and features matter more than model complexity for tabular data.
